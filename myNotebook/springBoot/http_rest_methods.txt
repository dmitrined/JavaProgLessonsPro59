2) HTTP и REST: виды запросов (методы)

В архитектуре REST (Representational State Transfer) взаимодействие между клиентом и сервером строится вокруг ресурсов, которыми мы манипулируем с помощью стандартных HTTP-методов.

2.1 Основные методы (CRUD-логика)

GET — чтение
Смысл: получить данные о ресурсе или коллекции ресурсов.
Характеристики:
- Тело запроса: ❌ Нет (данные передаются в URL или заголовках).
- Безопасность (Safe): ✅ Да (вызов метода не должен менять состояние сервера).
- Идемпотентность: ✅ Да (многократный повтор одного и того же запроса дает один и тот же результат).
- Кэшируемость: ✅ Да.
- Пример: `GET /api/cars/5` (получить машину с ID 5).

POST — создание
Смысл: создать новый ресурс или выполнить действие (команду), не вписывающуюся в CRUD.
Характеристики:
- Тело запроса: ✅ Да (обычно JSON).
- Safe: ❌ Нет (создает запись в БД).
- Идемпотентность: ❌ Нет (повторный POST создаст еще один ресурс).
- Пример: `POST /api/cars` (добавить новую машину в каталог).

PUT — полная замена
Смысл: заменить существующий ресурс целиком. Если ресурса нет, может создать его (зависит от логики).
Характеристики:
- Тело запроса: ✅ Да (полная модель объекта).
- Идемпотентность: ✅ Да (сколько бы раз мы ни отправляли одни и те же данные, ресурс примет именно это состояние).
- Пример: `PUT /api/cars/5` (обновить все поля машины №5).

PATCH — частичное обновление
Смысл: изменить только указанные поля ресурса.
Характеристики:
- Тело запроса: ✅ Да (только те поля, которые нужно изменить).
- Пример: `PATCH /api/cars/5` (изменить только цену или статус машины).

DELETE — удаление
Смысл: удалить ресурс.
Характеристики:
- Safe: ❌ Нет.
- Идемпотентность: ✅ Да (повторное удаление уже удаленного объекта не изменит состояние сервера — он все так же будет отсутствовать).
- Пример: `DELETE /api/cars/5` (удалить машину из базы).
